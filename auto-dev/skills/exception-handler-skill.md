# Exception Handler Skill

> 异常处理器 - 处理流水线中的各种异常
> 按需加载

---

## Skill 信息

| 属性 | 值 |
|------|-----|
| name | exception-handler |
| description | 异常处理器，负责处理开发、测试、部署各阶段的异常 |
| trigger | orchestrator自动调用 |

---

## 异常分类

### 按等级分类

| 等级 | 说明 | 影响 | 处理 |
|------|------|------|------|
| **P0** | 致命 | 阻塞整个流水线 | 立即停止，报告PM |
| **P1** | 严重 | 影响某个阶段 | 标记阻塞，协调解决 |
| **P2** | 一般 | 不影响流程 | 记录，继续执行 |

### 按阶段分类

| 阶段 | P0异常 | P1异常 |
|------|--------|--------|
| 需求分析 | 需求不明确无法推进 | 需求变更频繁 |
| 技术设计 | 技术方案不可行 | 设计遗漏某些场景 |
| 开发实现 | 编译失败 | 功能实现有误 |
| 测试验证 | 发现P0缺陷 | 发现P1缺陷 |
| 上线部署 | 部署完全失败 | 部分功能异常 |

## 异常处理规则

### 开发阶段异常

```markdown
## 编译失败 (P1)
处理流程：
1. 记录编译错误日志
2. 分析错误原因
3. 尝试自动修复（语法错误等）
4. 无法修复 → 通知backend-dev
5. 更新任务状态为blocked

## 代码逻辑错误 (P1)
处理流程：
1. 记录缺陷详情
2. 分析影响范围
3. 通知backend-dev修复
4. 修复后重新测试

## 技术难题阻塞 (P1)
处理流程：
1. 记录问题
2. 通知architect提供技术支持
3. 协调解决方案
4. 继续开发
```

### 测试阶段异常

```markdown
## 发现P0缺陷 (P0)
处理流程：
1. 立即停止测试
2. 记录缺陷详情
3. 通知backend-dev优先修复
4. PM决策是否继续
5. 修复后完整回归测试

## 发现P1缺陷 (P1)
处理流程：
1. 记录缺陷
2. 通知backend-dev修复
3. 修复后重新测试
4. 如果缺陷影响核心功能 → P0处理

## 测试环境异常 (P1)
处理流程：
1. 检查环境状态
2. 尝试重启服务
3. 无法解决 → 通知operation
4. 修复后继续测试
```

### 部署阶段异常

```markdown
## 部署失败 (P0)
处理流程：
1. 立即回滚（如有必要）
2. 记录错误日志
3. 分析失败原因
4. 尝试修复
5. 无法解决 → 报告PM
6. 修复后重新部署

## 健康检查失败 (P1)
处理流程：
1. 检查服务日志
2. 检查数据库连接
3. 检查依赖服务
4. 尝试修复
5. 无法解决 → 报告PM
```

## 阻塞任务处理

```markdown
当任务被阻塞时：

1. 创建异常记录
2. 更新任务状态为blocked
3. 尝试自动协调解决
4. 无法解决 → 报告PM
5. PM决策后继续执行
```

---

## 自动恢复规则

```markdown
| 异常类型 | 自动恢复尝试 |
|----------|--------------|
| 编译失败 | 最多重试3次 |
| 测试失败 | 最多重试3次 |
| 部署失败 | 最多重试2次 |
| 服务崩溃 | 自动重启 |

重试仍失败 → 标记blocked → 报告
```
